# manmaru-shiba CLAUDE.md

## このプロジェクトについて

柴犬のお世話を家族で共有するアプリ。ステップアップの転職活動に繋げる個人開発でもある。
技術選定の理由・設計判断を言語化して話せる状態にすることを重視する。

## ドキュメント作成のルール

### 開発・学習サイクル

```
① Claudeと会話して理解する（設計・技術選定の議論）
        ↓
② Claudeが実装・ドキュメント作成
        ↓
③ 自分でレビュー（GitHub PR・コードを読む）
        ↓
④ scrapの「言語化できるか」チェックリストを自分で潰す
        ↓
⑤ Zennスクラップに自分の言葉でアウトプット
```

①と④が「理解の入口と出口」。この2つが揃って初めて自分のものになる。

### 各成果物の役割

- **設計ドキュメント**（docs/design/）→ クリーンな仕様のみ
- **GitHub Issue** → 完了条件＋内容確認のレビューチェックリスト（「正しく動いているか」「抜け漏れがないか」）
- **GitHub PR** → 変更内容のサマリー＋内容確認のレビューチェックリスト
- **scrap**（docs/scrap/）→ 会話から抽出した作業ログ＋言語化チェックリスト（「自分の言葉で説明できるか」）

Issue/PRのレビューチェックリストは「内容の確認」に絞る。言語化・説明できるかの確認はscrapに書く。

ドキュメント作成後に「レビューしましたか？」等の確認は不要。
疑問が出たときにユーザーから聞いてくる。

### 作業ログの徹底

設計・実装の話がひと段落したら、その会話内容をもとに `docs/scrap/YYYY-MM-DD_content-in-english.md` を作成する。
ファイル名は必ず英語・ケバブケースにする（例：`2026-04-25_tech-stack-and-design-decisions.md`）。
ユーザーが改めて書き直す必要はない。会話から抽出して草案を作るのはClaudeの役割。

### レビュー時の言語化チェックフロー

PRを作成したあと、ユーザーがスクラップの言語化チェックリストを口頭で潰す作業を行う。

```
① ユーザーが各項目を口頭で説明する（Claudeなし・一人で）
        ↓
② Claudeにスクラップを見せてフィードバックをもらう（誤り・補足の確認）
        ↓
③ 理解が深まった内容・修正された認識をスクラップに追記
        ↓
④ 実装詳細（SDK・セキュリティ注意事項など）が出てきたらIssueにも追記
        ↓
⑤ ユーザー自身がコミット・push（このフェーズのコミットはユーザーのみ）
```

**スクラップの言語化チェックリストは回答も含めて記録する。**
チェックが終わったら `[ ]` を `[x]` に変え、各項目の下に回答を書き込む。


### 各成果物の更新タイミング

| 成果物 | 更新タイミング |
|---|---|
| スクラップ | 言語化チェック中・理解が深まったとき |
| Issue | 実装詳細・セキュリティ注意・SDK情報が決まったとき |
| 設計ドキュメント | 仕様の決定・変更があったとき |

## 言語化の視点

以下の観点で「数字や具体例で言語化できるか」を常に意識してレビューする：

- 技術選定の理由とトレードオフ（なぜAではなくBか）
- 設計の意図（なぜこうしたか・数字で説明できるか）
- 業務経験との接続（Javaでの非同期処理など）
- 競合との差別化

## 技術スタック

- バックエンド：Rails API only（Ruby）
- フロントエンド：Next.js（TypeScript）
- ジョブ処理：GoodJob（PostgreSQL使用・Redis不要）
- LINE連携：LIFF + Messaging API Webhook
- DB：PostgreSQL（Supabase）
- ホスティング：Render.com（Rails）/ Vercel（Next.js）

## ゼロ円構成を維持する

新しい依存を追加するときは必ず無料枠の確認を行う。
有料サービスを提案するときはその理由と代替案もセットで提示する。

## Git運用

- mainへの直pushは禁止
- 必ずブランチを切ってPRを作成する
- pushの前にユーザーに確認する
- PRマージ後はローカル・リモートともにブランチを削除する

### コミットの使い分け

- **Claudeがコミット** → Claudeが生成したドキュメント・実装コード
- **自分でコミット** → 言語化の回答・レビュー後の修正・自分が書いた内容

自分の言葉で書いた内容はClaude側でコミットしない。ユーザーに委ねる。
