# 開発ルール

**作成日：** 2026-04-25

---

## 1. ブランチ戦略

```
main        ← 常にデプロイ可能な状態
  └── feature/xxx   ← 機能ごとに切る
  └── fix/xxx       ← バグ修正
  └── docs/xxx      ← ドキュメントのみの変更
```

---

## 2. コミットメッセージ

```
feat: LINEからの排泄記録を追加
fix: 排泄アラートの時間計算がずれる問題を修正
docs: DB設計書を追加
refactor: WebhookJobの処理を整理
```

---

## 3. ドキュメントルール

### 作りっぱなし・実装しっぱなしを防ぐ

実装したら必ず `docs/scrap/` に作業ログを残す。

```
docs/scrap/
└── YYYY-MM-DD_やったこと.md
```

**書く内容（雑でOK）：**
- 何を実装したか
- なぜこう設計したか
- 詰まったこととどう解決したか
- 面接で話せそうなポイント

### アウトプットの流れ

```
docs/scrap/（雑なメモ）
    ↓ 整理
Zennスクラップ（公開アウトプット）
    ↓ 面接ネタに絞る
~/Private/Career/（言語化・面接で話せる形）
```

---

## 4. ディレクトリ構成

```
manmaru-shiba/
├── backend/        # Rails API only
├── frontend/       # Next.js
└── docs/
    ├── design/     # 要件定義・DB設計・API設計
    ├── rules/      # 開発ルール（このファイル）
    └── scrap/      # 作業ログ・実装メモ
```

---

## 5. ゼロ円構成の維持

新しいサービス・ライブラリを追加するときは、無料枠を超えないか確認する。

| サービス | 無料枠の注意点 |
|---|---|
| Render.com | 15分スリープあり → UptimeRobotでping |
| Supabase | 500MB・50,000行まで |
| Vercel | 個人利用は基本無料 |
| LINE Messaging API | 月1,000通まで無料 |
