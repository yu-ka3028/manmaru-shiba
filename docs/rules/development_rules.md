# 開発ルール

**作成日：** 2026-04-25

---

## 1. ブランチ戦略

```
main        ← 常にデプロイ可能な状態（直push禁止）
  └── feature/xxx   ← 機能ごとに切る
  └── fix/xxx       ← バグ修正
  └── docs/xxx      ← ドキュメントのみの変更
```

**mainへの直pushは禁止。必ずブランチを切ってPRを作成する。**
ドキュメント追加・修正も`docs/xxx`ブランチを切ってPR経由で入れる。

---

## 2. コミットメッセージ

```
feat: LINEからの排泄記録を追加
fix: 排泄アラートの時間計算がずれる問題を修正
docs: DB設計書を追加
refactor: WebhookJobの処理を整理
```

---

## 3. ドキュメントルール

### 作りっぱなし・実装しっぱなしを防ぐ

実装したら必ず `docs/scrap/` に作業ログを残す。

```
docs/scrap/
└── YYYY-MM-DD_やったこと.md
```

**書く内容（雑でOK）：**
- 何を実装したか
- なぜこう設計したか
- 詰まったこととどう解決したか
- 言語化しておきたいポイント

### スクラップのテンプレート

````markdown
# テーマのタイトル

**日付：** YYYY-MM-DD
**テーマ：** 何についての議論か

---

## 決定事項まとめ
（設計・技術選定の結論を箇条書き）

## 却下した案と理由
| 案 | 却下理由 |
|---|---|

## 用語整理
（新しく出てきた概念・技術をまとめる）

---

## 言語化チェックリスト

- [ ] 項目1

  （レビュー後に回答を書き込む）

- [ ] 項目2

  （レビュー後に回答を書き込む）
````

**言語化チェックリストはレビュー時に口頭で潰し、回答をそのまま書き込む。**
チェックが終わったら `[ ]` を `[x]` に更新する。

### アウトプットの流れ

```
docs/scrap/（雑なメモ）
    ↓ 整理
Zennスクラップ（公開アウトプット）
    ↓ 言語化しておきたいポイントに絞る
~/Private/Career/（言語化できる形）
```

## 4. ディレクトリ構成

```
manmaru-shiba/
├── backend/        # Rails API only
├── frontend/       # Next.js
└── docs/
    ├── design/     # 要件定義・DB設計・API設計
    ├── rules/      # 開発ルール（このファイル）
    └── scrap/      # 作業ログ・実装メモ
```

---

## 5. ゼロ円構成の維持

新しいサービス・ライブラリを追加するときは、無料枠を超えないか確認する。

| サービス | 無料枠の注意点 |
|---|---|
| Render.com | 15分スリープあり → UptimeRobotでping |
| Supabase | 500MB・50,000行まで |
| Vercel | 個人利用は基本無料 |
| LINE Messaging API | 月1,000通まで無料 |
